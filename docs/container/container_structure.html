コンテナの構成

コンテナは単一のホストOS上で動作するコンテナエンジン上にプロセスとして存在し，そのエンジンが管理するリソースの範囲内で，複数のコンテナを同時に動作させることができる．
コンテナ内には、”アプリケーション（＝プロセス）”と、その実行に必要な”ミドルウェア”，”ライブラリ”等が含まれる．このライブラリの中にはOSの基本的なコマンドセットやファイルシステムのライブラリ等，OSが提供する機能の一部が含まれている．

画像

様々なLinuxOSのコンテナイメージが公開されており，これらを用いることでアプリケーションの実行環境を構築できる．しかし，コンテナはOS自体を動作させているわけではなく，それに似た動作環境を再現している．その理解の為に，コンテナにおけるアプリケーションの起動方法を解説する．
コンテナ内のアプリケーションは、コンテナ内のミドルウェアやライブラリを通して、ホストOSのカーネルのシステムコールを呼び出すことで動作している．ホストOSのカーネルがシステムコールを処理することでアプリケーションを動作させている．実際には ABI(Application Binary Interface) と呼ばれる専用のインターフェース経由でやり取りされる．

コンテナの種類

コンテナには大きく分けて，２種類のものがある．

アプリケーションコンテナ

アプリケーションコンテナとは，１つのサービスをパッケージ化して実行するように設計されたコンテナである．１コンテナにつき１サービスであるため，実行が軽い，コンテナ内のアプリケーションの依存関係がシンプルになる，といった特徴がある．アプリケーションコンテナの例としては，DockerやRocketがある．
また，大きな特徴の１つとしては，他のコンテナ技術の上に構築できる点である．アプリケーションコンテナの背後にある考え方は，アプリケーション内のコンポーネントごとに異なるコンテナを作成するというものである．このアプローチは，マイクロサービスアーキテクチャを使用して，分散型のマルチコンポーネントシステムをデプロイする場合に特に有効である．

システムコンテナ

システムコンテナとは，アプリケーションコンテナと異なり，複数のプロセスとサービスが動作し，実際のLinux OSと同様に動作するコンテナである．OSを丸ごとコンテナ内で実行するようなイメージである．その為，他のOSと同じように，アプリケーションやライブラリ等をインストール，構成，実行することが出来る．なお，VMと同じように，コンテナ内で実行されているものはすべて，そのコンテナに割り当てられているリソースのみを表示する．システムコンテナの例としては，LXCやOpenVZがある．

画像

従って，一般に，アプリケーションをコンポーネントとしてパッケージ化して配布する場合は，アプリケーションコンテナが適切な手段として機能する．一方で，様々なライブラリや言語，DBなどをインストールできるOSが必要な場合は，システムコンテナの方が適している．
